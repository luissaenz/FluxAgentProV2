## 2. Plan Realista (basado en lo que falta)

En lugar de crear un sistema de tickets completo (como propone el doc), vamos a implementar algo más simple que use la infraestructura existente:

### Enfoque: Crear tarea directamente desde UI

1. **Backend: POST /tasks** — Endpoint simple para crear tarea y ejecutar flow
2. **Backend: GET /flows** — Listar flows del registry  
3. **Frontend: Botón "Nueva Tarea"** — En página /tasks
4. **Frontend: Página /tasks/new** — Formulario simple

Esto es más rápido de implementar y cumple el mismo objetivo (permitir al usuario levantar trabajo).

## 2. Feature 1: Levantar Tickets (botón + endpoint POST /tasks)

### Backend: Endpoint POST /tasks

El endpoint `/webhooks/trigger` ya existe y hace lo mismo, pero usa otro naming. Vamos a crear un endpoint POST en `/tasks` que sea más intuitivo:

```python
# src/api/routes/tasks.py - AGREGAR después de los GETs

class CreateTaskRequest(BaseModel):
    flow_type: str
    input_data: dict = {}
    correlation_id: Optional[str] = None

@router.post("", status_code=202)
async def create_task(
    request: CreateTaskRequest,
    background_tasks: BackgroundTasks,
    auth: dict = Depends(verify_org_membership),
):
    """Create a new task and execute the flow in background."""
    org_id = auth["org_id"]
    
    # Validar que el flow existe en registry
    if not flow_registry.has(request.flow_type):
        raise HTTPException(400, f"Flow '{request.flow_type}' no encontrado")
    
    from uuid import uuid4
    from .webhooks import _run_async_in_background, execute_flow
    
    task_id = str(uuid4())
    correlation_id = request.correlation_id or str(uuid4())
    
    # Crear task en DB
    with get_tenant_client(org_id) as db:
        db.table("tasks").insert({
            "id": task_id,
            "org_id": org_id,
            "flow_type": request.flow_type,
            "status": "pending",
            "input_data": request.input_data,
            "correlation_id": correlation_id,
        }).execute()
    
    # Ejecutar en background
    background_tasks.add_task(
        _run_async_in_background,
        execute_flow(request.flow_type, org_id, request.input_data, correlation_id),
    )
    
    return {"task_id": task_id, "correlation_id": correlation_id, "status": "accepted"}
```

## 8. Notas técnicas

- **Flujo existente:** Los bartenders flows tienen endpoints específicos (`/bartenders/preventa`, etc.) — el nuevo `/tasks` es genérico y cubre todos los flows
- **Input validation:** Validar input_data contra schema del flow (opcional para MVP)
- **Transcription:** Los eventos en domain_events ya tienen payload — usarlos para construir el timeline
- **Panel de agente en FAP:** FAP no tiene "agentes" como Paperclip. El "panel de agente" equivalente es la página de detalle de tarea (`/tasks/[id]`) que muestra el estado, eventos, y resultado del flow.
- **Credenciales/Vault:** Por ahora, las credenciales no se exponen en la UI — el Vault es solo backend. Esto se traite en semanas futuras.
